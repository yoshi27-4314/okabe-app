# OKABE GROUP タイヤホテル業務管理アプリ — ゼロからの再設計プロンプト

## あなたへの依頼

私はファーストエイトグループ代表の浅野です。OKABE GROUPのタイヤホテル事業向け業務管理アプリを、計画段階からゼロベースで作り直したいと考えています。

最終成果物は「Claude Codeにそのまま渡せる設計書一式」です。からだ記録アプリで成功した「建築発想」の設計手法を踏襲してください。

## 建築発想の設計手法（からだ記録で確立した方法論）

からだ記録アプリでは、建築の考え方でソフトウェアを設計しました。この手法をokabe-appにも適用します。

### ドキュメント構成（7つの設計図）

```
00_DECISIONS.md      決定済み事項の記録（施主の意思決定の履歴）
01_MANIFESTO.md      思想（なぜ建てるのか、どんな建物か）
02_UX_DESIGN.md      使う人の体験（平面図、動線設計）
03_DATA_SCHEMA.md    データ構造（構造図、骨組み）
04_SYSTEM_SPEC.md    システム仕様（設備の挙動仕様）※からだ記録のAI_BEHAVIORに相当
05_TECH_SPEC.md      技術仕様（材料・工法の指定）
06_BUILD_PLAN.md     施工計画（工程表、組み立て順序）
```

### この手法の核心ルール

1. **設計図は最初から完成形を作る** — 段階リリース(MVP)の考え方は採用しない
2. **実装の順序は優先順位ではなく依存関係で決まる** — 「何が重要か」ではなく「何が先にないと次が作れないか」
3. **UIは最初から完成形を提示する** — 段階的に画面を増やさない
4. **エラーが起きたら3分類で特定する** — 設計ミス・計画ミス・作業ミスのいずれか
5. **設計と工程は分ける** — 「いつ作るか」ではなく「どう作るか」

## 現行アプリの完全な分析

### 事業概要

- **事業**: タイヤホテル（タイヤの預かり・保管・履き替えサービス）
- **運営**: 株式会社OKABE GROUP（岐阜県岐阜市六条大溝）
- **社長**: 岡部社長
- **対象顧客**: 個人・法人（運送会社、建設会社等）
- **主な業務フロー**: 顧客受付 → タイヤ預かり → 保管 → 履き替え作業 → 返却

### 現行の技術構成

- HTML/CSS/JavaScript（フレームワークなし）
- GitHub Pages でホスティング（https://yoshi27-4314.github.io/okabe-app/）
- Supabase（PostgreSQL + Auth）— プロジェクトref: peportftucwuxfnmaanr
- Supabase Edge Function（tire-ocr）— Claude Vision APIでOCR処理
- PWA対応（manifest.json あり）

### 現行のファイル構成と問題点

```
index.html      (315行) ログイン + 6ロール別ホーム画面
demo.html       (427行) デモ用（ロール選択カード付き）— index.htmlとほぼ重複
staff.html      (1071行) 全画面を1ファイルに詰め込み ← 最大の問題
demo-staff.html (2131行) デモ用staff — staff.htmlとほぼ重複
css/common.css  (435行) 共通CSS + エルメス/グッチテーマ
css/staff.css   スタッフ画面用CSS
js/supabase.js  (163行) DB操作関数群
```

### 構造上の問題

1. **巨大な1ファイル**: staff.htmlに全画面（12画面）が詰め込まれている
   - ホーム、作業一覧、顧客情報、新規登録(4ステップ)、検索、チャットBot、預かり台帳、約款、車検証読取、発注希望、発注一覧、納品チェック
2. **デモと本番の重複**: demo.html/demo-staff.htmlがindex.html/staff.htmlのコピー
3. **alertだらけ**: 売上・設定・通知・スタッフ状況など多数のボタンがalert()のまま未実装
4. **CSSの散在**: css/フォルダとHTMLインラインスタイルに分散
5. **画面遷移が不自然**: index.html → staff.html?tab=xxx のパラメータ遷移

### 現行の6ロールと機能マップ

| ロール | ID | 主な機能 | 実装状況 |
|--------|-----|----------|----------|
| 岡部社長 | president | ダッシュボード、売上、スタッフ状況、全体把握 | ダッシュボードのみ。売上・スタッフはalert() |
| 営業部長 | sales_manager | 営業チーム管理、交換推奨フォロー、売上 | ホーム画面のみ。チーム管理はalert() |
| 営業 | sales | 担当顧客、訪問記録、交換推奨 | ホーム画面のみ。訪問記録はalert() |
| 工場長 | factory_manager | 作業管理、判定確認、スタッフ状況 | ホーム画面のみ。判定確認はalert() |
| フロント事務 | office | 予約受付、顧客管理、台帳、発注管理、約款、エクスポート | 最も機能が充実。発注書3社分の帳票生成あり |
| サービススタッフ | service | 作業一覧、ナンバー読取、発注希望送信、納品チェック | OCR連携・納品照合まで実装済み |

### 現行のDBテーブル構成（Supabase）

```
staff           — スタッフ(auth_user_id, name, role)
customers       — 顧客(name, kana, type[個人/法人], phone, mobile, address, line_id, web_registered, customer_status)
vehicles        — 車両(customer_id, plate_number, maker, model, color)
tire_sets       — タイヤセット(vehicle_id, season_type, storage_location_no, fee)
tires           — タイヤ個別(tire_set_id, position[左前/右前/左後/右後], manufacturer, pattern, size, tread_depth, needs_replacement, wheel_type, wheel_material, hubcap, lug_nuts, inspector)
storage_periods — 預かり期間(tire_set_id, start_date, planned_return_date, status[預かり中/期限超過/返却済])
works           — 作業(vehicle_id, tire_set_id, type, status[予約/受付/作業中/完了/キャンセル], scheduled_date, completed_date, memo)
photos          — 写真(tire_set_id)
settings        — 設定(key, value) — 残溝判定閾値など
terms           — 約款
```

### 実装済みの機能（動作するもの）

1. **ログイン/認証**: Supabase Auth + staffテーブルでロール判定
2. **作業一覧**: 日付フィルター(今日/明日/今週)、ステータス別グループ表示、ステータス変更(予約→受付→作業中→完了)
3. **顧客管理**: 検索(名前/電話/ナンバー)、顧客詳細表示(車両・タイヤセット・タイヤ4本の情報)
4. **新規登録**: 4ステップウィザード(顧客→車両→タイヤセット→タイヤ4本)
5. **預かり台帳**: フィルター(全て/預かり中/期限超過/返却済)、期限超過の自動検出・更新
6. **OCR/カメラ**: ナンバー読取、タイヤ情報読取、残溝測定、車検証読取、納品書読取（全てSupabase Edge Function経由のClaude Vision API）
7. **車検証→新規登録**: 車検証をOCRで読み取り→フォームに自動入力→確認チェック
8. **発注管理**: サービススタッフが発注希望を送信→フロント事務が受け取り→発注先選択(MS.トーカイ/ブリヂストンはきの/フジ・コーポレーション)→各社書式の発注書を生成→印刷
9. **納品チェック**: 納品書スキャン + タイヤ実物スキャン → メーカー/パターン/サイズの照合結果表示
10. **残溝判定**: 設定テーブルの閾値に基づく4段階判定(良好/要注意/交換推奨/使用不可)
11. **約款表示**: タイヤ保管同意書（7条）

## 打ち合わせで決めたいこと

### 1. 事業の本質と思想（MANIFESTO）

- このアプリは「何」なのか？（ただの業務ツール？ 岡部社長の経営の目？ 現場の作業効率化？）
- 岡部社長にとっての価値は何か？
- 現場スタッフにとっての価値は何か？
- 顧客にとっての価値は何か（将来的に顧客向け画面を作る可能性は？）

### 2. ロール構成の見直し

- 6ロールは多すぎないか？ 実際に使う人は何人？
- 岡部社長が本当に見たい情報は何か？（数字？ 作業状況？ 異常？）
- 営業部長と営業は分ける必要があるか？
- 工場長とサービススタッフの違いは何か？

### 3. 業務フローの整理（UX DESIGN）

- タイヤ1セットの「受付→預かり→保管→履き替え→返却」の完全な流れ
- 誰が・いつ・何をするか（業務動線）
- 現場で最も使う画面は何か？
- 最も時間がかかっている作業は何か？
- 紙の帳票はまだ使っているか？（発注書以外）

### 4. データ構造の見直し（DATA SCHEMA）

- 現行テーブルで足りないもの・不要なもの
- 顧客の車両が増えた時の対応
- 履歴管理（過去の預かり・作業の履歴）
- 写真の保存先（Supabase Storage? Google Drive?）

### 5. 技術選定（TECH SPEC）

以下の選択肢がある：

| 選択肢 | メリット | デメリット |
|--------|----------|------------|
| A: 現行と同じHTML/JS | シンプル、GitHub Pagesそのまま | ファイル肥大化が再発する |
| B: Next.js + Supabase + Vercel | からだ記録と同じ構成。コンポーネント分割が自然 | 学習コスト、Vercelのプラン |
| C: React SPA(Vite) + GitHub Pages | フレームワークの恩恵 + 無料ホスティング | SSR不要ならこれで十分 |

- からだ記録のSupabaseプロジェクト(njdnfvlucwasrafoepmu)と共用するか、okabe専用(peportftucwuxfnmaanr)を継続するか
- Edge Function(OCR)はそのまま活かせるか

### 6. UIテーマの方向性

- エルメス×グッチの高級テーマを継続するか
- もっとシンプルで実用的な方向にするか
- 岡部社長の好みは把握しているか

### 7. デモモードの扱い

- 本番とデモを統合するか（「デモログイン」ボタン1つ）
- それとも別URLでデモ版を残すか
- 岡部社長へのプレゼンはどうやっているか

## 打ち合わせの進め方

1つずつ、対話しながら決めていきたいです。全部を一度に決める必要はありません。

**進め方の提案**:
1. まず事業の本質（MANIFESTO）を固める
2. 次に業務フローと動線（UX DESIGN）を整理する
3. そこからデータ構造（DATA SCHEMA）が決まる
4. 技術選定（TECH SPEC）は最後でいい
5. 各段階で00_DECISIONSに決定事項を記録していく

最初の質問から始めてください。「このアプリは、岡部社長と現場スタッフにとって、何であるべきですか？」
